# 服务端网络通信扩展计划

## 1. 需求目标
当前实验需要在现有服务端界面项目基础上，继续完善网络通信能力，使服务端在启动后能够接收并处理客户端发送的数据包，覆盖以下典型场景：
- 客户端登录：客户端发送用户名和密码，服务端校验身份是否合法
- 客户端注销：服务端接收注销请求，并通知其他客户端更新在线用户列表
- 客户端聊天：客户端之间的聊天消息通过服务器进行转发
- 日志展示：服务端界面中的日志输出需要做颜色标记，提升可读性
- 验证运行：完成后能够运行项目并进行测试

## 2. 当前项目完成情况检查

### 已完成
1. 已具备服务端启动能力
   - `server_app/network/server_controller.py` 已实现基于 `socketserver.ThreadingTCPServer` 的 TCP 服务启动与停止
   - `server_app/ui/main_window.py` 已通过启动按钮触发服务启动/停止

2. 已具备登录校验基础能力
   - 服务端已能接收 `LOGIN <username> <password>` 命令
   - 调用 `Database.verify_login()` 校验用户名和密码
   - 数据库存储的是带盐 PBKDF2 哈希，不是明文密码

3. 已具备注销请求接收基础能力
   - 已支持 `LOGOUT <username>` 命令
   - 已能记录注销日志并回写 `OK`

4. 已具备基础日志输出能力
   - 主窗口中已有日志面板
   - 当前通过 `append_log()` 以纯文本方式输出日志

## 部分完成
1. 服务端网络通信已启动，但协议能力较弱
   - 当前仅支持简单文本协议
   - 未形成统一的数据包结构

2. 多线程接入已具备基础
   - 使用 `ThreadingTCPServer`，每个连接由独立线程处理
   - 但尚未实现完整的连接状态管理

## 未完成
1. 在线用户管理未完成
   - 当前没有真正维护在线客户端连接映射
   - 注销时也没有广播其他客户端更新在线用户列表

2. 消息转发未完成
   - 当前没有聊天消息命令
   - 没有点对点或广播消息转发机制

3. 日志颜色标记未完成
   - 当前日志控件是 `QPlainTextEdit`
   - 仅支持纯文本追加，不支持按消息类别着色

4. 自动化测试未完成
   - 当前仓库没有发现现成的 pytest/unittest 测试用例
   - 运行环境目前也缺少 `PyQt5` 依赖，项目尚无法直接启动

## 3. 建议实施方案

### 阶段一：日志颜色标记
目标：优先完成一个对现有界面侵入较小、可快速验证的需求。

建议做法：
- 将日志控件从 `QPlainTextEdit` 调整为支持富文本的控件
- 按日志类别着色，例如：
  - 成功：绿色
  - 失败/错误：红色
  - 警告：橙色
  - 普通信息：黑色或深灰色
  - 网络事件：蓝色
- 封装统一日志输出方法，例如按消息内容或日志级别决定颜色

### 阶段二：在线客户端状态管理
目标：让服务端知道“谁在线”。

建议做法：
- 在 `ServerController` 中维护在线用户表
  - 用户名 -> 客户端连接/会话信息
- 登录成功后登记在线用户
- 注销或连接断开后移除在线用户
- 统一封装在线状态广播方法

### 阶段三：扩展协议与广播能力
目标：支持客户端列表更新和聊天消息转发。

建议协议方向：
- `LOGIN <username> <password>`：登录
- `LOGOUT <username>`：注销
- `MSG <from> <to> <content>`：私聊消息转发
- `BROADCAST_USERS ...`：服务端通知在线用户变化

建议注意点：
- 当前项目是简单文本协议，需考虑空格分割后消息内容的保留问题
- 可以先保持行协议，再约定固定字段顺序
- 后续若需要更稳妥，可升级为 JSON 行协议

### 阶段四：联调与测试
目标：验证网络逻辑和界面日志显示。

建议测试项：
1. 服务端能否正常启动监听
2. 正确账号登录返回成功
3. 错误账号登录返回失败
4. 用户注销后是否触发在线列表更新通知
5. 消息发送后是否被服务器正确转发
6. 日志颜色是否符合预期
