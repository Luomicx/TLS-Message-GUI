# 客户端架构审查与 UI 布局优化计划

## 1. 目标

本计划面向当前 `client_app` 的实际代码结构，目标不是重写客户端，而是在保持现有 PyQt5 + QFluentWidgets 技术路线不变的前提下，完成一轮低风险、高收益的客户端 UI 与架构整理。重点聚焦以下两类问题：

- 客户端界面布局层级不够清晰，部分页面仍有明显占位式结构
- `client_app` 的 UI 状态、业务编排、数据展示规则已开始互相耦合，继续扩展会抬高维护成本

本计划优先处理：

1. 聊天窗口的布局与上下文保持问题
2. 搜索页的布局与动作语义问题
3. 登录页的主次信息层级问题
4. 好友页的结构扩展性问题
5. 客户端 UI 与应用编排层之间的边界整理

## 2. 现状概览

### 2.1 当前客户端架构

当前 `client_app` 基本形成了以下结构：

- `client_app/app.py`
  - `ClientApplication` 负责应用级编排
  - 连接 UI signal、调用 `ClientController`、切换窗口、映射错误消息
- `client_app/network/client_controller.py`
  - 负责 TLS socket 建链、同步请求/响应、JSON line 协议传输
- `client_app/protocol.py`
  - 负责请求编码与响应解码
- `client_app/ui/`
  - `login_window.py`
  - `register_dialog.py`
  - `chat_window.py`
  - `theme.py`

整体分层方向是正确的：

- UI 不直接处理 socket
- 应用层负责接线与流程编排
- 网络传输层与协议层独立存在

### 2.2 当前主要窗口结构

#### 登录窗口 `LoginWindow`

- 使用左右双栏布局
- 左侧为品牌与说明区
- 右侧为登录表单区
- 默认窗口尺寸 `1080x720`
- 最小尺寸 `920x620`

#### 注册弹窗 `RegisterDialog`

- 基于 `MessageBoxBase`
- 采用单列垂直表单结构
- 包含用户名、密码、确认密码、勾选项
- 当前主要依赖最小宽度保护与 `InfoBar` 提示

#### 聊天主窗 `ChatWindow`

- 基于 `FluentWindow`
- 当前包含 3 个子页：
  - 聊天页
  - 好友页
  - 搜索页
- 聊天页为左右双栏：
  - 左：账号信息 + 会话列表 + 注销按钮
  - 右：聊天头部 + 消息区 + 输入区

## 3. 架构审查结论

### 3.1 优点

当前客户端已有以下结构优势：

- 顶层分层清晰：`ui / app / network / protocol`
- `ClientApplication` 统一管理窗口切换和业务流程
- `ClientController` 已形成稳定的同步 request/response 传输接口
- UI 主要通过 signal 发出意图，而不是直接发网络请求
- 主题与字号相关逻辑已经抽出到 `theme.py`

### 3.2 主要问题

#### 问题一：`ChatWindow` 过大且职责混合

`client_app/ui/chat_window.py` 同时承担：

- 页面布局构建
- UI 局部状态存储
- 展示文本拼装
- 列表重建逻辑
- 搜索与发送消息交互前置校验
- demo/mock 初始化痕迹

这会导致：

- 后续功能继续堆入同一个类
- 布局调整与数据展示规则互相牵连
- 测试难度上升

#### 问题二：`ClientApplication` 开始变胖

`client_app/app.py` 中的 `ClientApplication` 除了接线外，还承担：

- 错误码映射
- 服务端消息到 UI 数据的转换
- 当前用户状态管理
- 登录/注销/搜索/加好友/发消息/拉消息的全部流程协调

当前规模下尚可接受，但已接近需要拆出辅助层的临界点。

#### 问题三：UI 对外接口风格不统一

当前存在三种不同模式：

- signal：登录、搜索、发消息、切换会话
- callable 注入：注册对话框 `register_submitter`
- 直接操作内部控件：`chat_window.btn_logout.clicked`

这会让 UI 边界不统一，后续维护和复用成本上升。

#### 问题四：同步网络调用仍在 UI 主线程内执行

`ClientController` 当前是同步阻塞式网络调用：

- `socket.create_connection()`
- `sendall()`
- `recv()`

它虽然继承 `QObject` 并暴露多个 signal，但当前主流程并未真正使用这些 signal 作为异步通信机制。结构上呈现出“外观像异步，实际是同步 service”的混合状态。

#### 问题五：会话列表与好友列表存在边界混叠

`ChatWindow.populate_friends()` 会同时：

- 清空好友列表
- 清空会话列表
- 重建两者
- 默认把当前会话切到第一个联系人

这意味着：

- 刷新好友会影响当前聊天上下文
- 好友列表与最近会话列表的 UI 责任混在一起

## 4. 当前布局问题分析

### 4.1 登录页

当前登录页采用左右双栏，视觉方向是合理的，但存在以下问题：

- 左侧 hero 区信息量较多，和右侧登录区抢主视觉
- 登录按钮和访客预览按钮同级，不利于强调主任务
- 登录失败、输入错误等提示全部依赖 `InfoBar`，缺少页内固定状态区
- 在较窄窗口下，hero 区仍占用较多注意力

### 4.2 注册弹窗

当前注册弹窗整体可用，但层级还不够成熟：

- 表单说明区、字段区、确认区的分界不强
- 错误提示全部浮动显示，缺少页内固定错误锚点
- 宽度保护主要依赖硬性的 `minimumWidth`

### 4.3 聊天页

聊天页的“左会话、右聊天”骨架是正确的，但存在以下问题：

- sidebar 宽度策略偏硬，只靠 `280~340` 做限制
- 侧栏 profile 信息仍有明显占位文案，且与聊天头部信息不统一
- 输入区高度可用，但在小窗口下空间利用率不够灵活
- 刷新好友/会话数据时会打断当前用户阅读上下文

### 4.4 好友页

好友页当前基本是：

- 标题
- 描述
- 列表

虽然可用，但更像“占位式页面”，没有形成清晰的“摘要区 + 内容区”结构，也不利于后续加入筛选、在线状态统计、空态、选中详情等能力。

### 4.5 搜索页

搜索页当前最大的问题不是组件不够，而是布局没有清楚表达交互流程：

- 搜索输入、搜索模式和按钮都在同一行
- 主按钮文案是“添加好友”，但在没有选中搜索结果时它实际触发的是搜索
- 页面缺少“结果状态区”，无法清晰表达未搜索、无结果、有结果等不同状态

## 5. 优化原则

本轮优化遵循以下原则：

### 5.1 低风险优先

- 不引入新依赖
- 不改动现有网络协议
- 不推翻 `ClientApplication` 作为应用编排层的定位
- 不把当前 UI 全部重写成新组件系统

### 5.2 布局先行，业务其次

本计划主要解决：

- 信息层级
- 页面分区
- 窗口缩放行为
- 用户上下文保持

不在本轮内追求：

- 全量消息气泡重做
- 大规模动画与视觉特效
- 复杂富文本或自定义 item delegate

### 5.3 保留用户上下文

任何刷新列表、切换页签、更新状态的优化都应优先保证：

- 当前会话不被意外切走
- 当前好友/搜索结果选中项尽量保留
- 消息阅读上下文不被破坏

## 6. 分页面优化方案

### 6.1 登录页优化

目标：让“进入聊天”成为明确主任务，同时保留现有 Fluent 风格。

建议：

1. 保留双栏结构，但弱化左侧 hero 权重
2. 调整左右区域比例，使登录卡片更突出
3. 将“进入聊天”提升为唯一主操作
4. 将“访客预览”降为次级操作
5. 在登录表单区增加页内固定状态区，用于显示校验或登录失败提示
6. 在接近最小宽度时折叠 hero 区的辅助文案或部分标签项

### 6.2 注册弹窗优化

目标：保持轻量表单弹窗定位，提高信息层级和错误定位效率。

建议：

1. 明确三段式层级：说明区、字段区、确认区
2. 增加页内固定错误区，而不是全部依赖 `InfoBar`
3. 放宽局部字段块的最小宽度约束，减少字号变大后的拥挤感
4. 保持“完成注册”为主按钮，“返回登录”为次按钮

### 6.3 聊天页优化

目标：提升聊天页面的空间分配质量与状态一致性，同时修复最明显的用户上下文问题。

建议：

1. 保留左右双栏骨架，不做激进结构重写
2. 优化 sidebar 宽度策略，使其更柔性，不只靠固定 `maxWidth`
3. 压缩或重构 sidebar 顶部资料卡，使会话区获得更多可用高度
4. 保持“头部 / 消息区 / 输入区”三段式，但提高输入区高度的弹性
5. 刷新好友或会话时优先恢复原会话，而不是默认切回第一项
6. 统一 sidebar 资料区与聊天头部的用户信息来源，避免一处更新、一处占位

### 6.4 好友页优化

目标：把好友页从“占位式单列表”提升为“可扩展的联系人管理页”。

建议：

1. 拆成“顶部摘要区 + 列表区”两段结构
2. 顶部摘要区可承载：
   - 好友数量
   - 在线/离线概览
   - 轻量筛选入口预留
3. 保持列表区为主伸缩区域
4. 刷新列表时尽量保留用户当前选择

### 6.5 搜索页优化

目标：让“输入条件 -> 搜索 -> 查看结果 -> 添加好友”的流程在布局上直接可见。

建议：

1. 将顶部由单行工具栏升级为更清晰的搜索控制区
2. 分离“搜索”与“添加好友”两个动作
3. 仅在选中搜索结果时启用“添加好友”按钮
4. 在结果列表上方增加结果状态区，用于表达：
   - 未搜索
   - 搜索中
   - 无结果
   - 有结果
5. 刷新搜索结果时尽量保留当前结果上下文，不要强制打断操作流程

## 7. 架构整理建议

本轮不做重构式改造，但建议同步进行以下低扰动整理：

### 7.1 清理 `ChatWindow` 的 demo/mock 痕迹

建议逐步移除或隔离：

- `_load_mock_data()`
- `_mock_send_message()` 命名与初始化逻辑
- 页面中的“占位”“后续接入真实数据”等文案残留

正式运行路径与演示预览路径应分离。

### 7.2 统一 UI 对外接口风格

建议逐步统一为：

- 窗口发 signal
- `ClientApplication` 作为唯一业务编排层接收 signal

优先处理：

- 注册弹窗从 `register_submitter` 过渡到 signal 风格
- 聊天窗口注销行为从直接暴露按钮，过渡为窗口级 signal

### 7.3 收敛展示映射逻辑

建议将以下逻辑逐步从 `ClientApplication` 和 `ChatWindow` 中抽为辅助函数：

- 错误码到用户文案的映射
- 服务端消息到 UI 消息结构的转换
- 在线状态、会话头部、搜索结果项的展示文案拼装

### 7.4 重新定义好友列表与会话列表的边界

建议避免再使用一个方法同时维护：

- 好友列表
- 最近会话列表

后续应明确：

- `populate_friends()` 只刷新好友列表
- 会话列表使用单独的刷新入口

## 8. 分阶段实施建议

### 阶段一：低风险交互与布局整理（优先）

目标：在不改协议、不改传输层的情况下，先修最明显的体验问题。

任务：

1. 修复聊天窗口刷新后丢失当前会话的问题
2. 搜索页拆分“搜索”与“添加好友”两个动作
3. 登录页增加页内状态区，重排主次按钮
4. 好友页加入顶部摘要区

### 阶段二：页面结构增强

目标：提升页面层级清晰度和缩放稳定性。

任务：

1. 优化登录页窄窗口下的 hero 内容折叠策略
2. 优化聊天页侧栏宽度与输入区高度弹性
3. 统一聊天页头部与侧栏 profile 信息来源
4. 为搜索页和好友页预留扩展槽位

### 阶段三：边界清理与结构收敛

目标：在不进行大规模重构的前提下，减少 UI 和应用层的耦合噪音。

任务：

1. 清理 `ChatWindow` 中的 demo/mock 初始化残留
2. 统一 UI 组件对外接口风格
3. 抽离消息映射与错误文案解析辅助函数
4. 重新划分好友与会话的 UI 刷新职责

## 9. 验证标准

### 9.1 登录页

- 在最小窗口尺寸下，登录字段和主按钮仍清晰可用
- 主按钮明显强于访客预览
- 登录失败信息能够在表单附近直接看到

### 9.2 注册弹窗

- 大字号下字段区仍保持清晰节奏
- 错误提示能够在表单上下文中定位
- 按钮区无拥挤或被遮挡现象

### 9.3 聊天页

- 在当前最小窗口尺寸下，消息区仍为主视觉区域
- 刷新好友或会话时，不再无条件跳到第一会话
- 当前聊天对象信息在 sidebar 与头部保持一致

### 9.4 好友页

- 页面具备清晰的摘要区和列表区
- 列表仍是主伸缩区域
- 后续可自然扩展状态或筛选信息

### 9.5 搜索页

- 搜索和添加好友两个动作职责清晰
- 页面能明确区分未搜索、无结果、有结果三种状态
- 添加好友后不会强制打断当前搜索上下文

### 9.6 架构层

- 不引入新依赖
- 不破坏 `ClientApplication` 现有主流程
- 不破坏现有 signal 命名与网络接口兼容性

## 10. 结论

当前 `client_app` 的整体分层方向是正确的，但已处于“继续堆功能会显著增加维护成本”的阶段。最需要优先处理的不是网络协议，而是客户端内部的 UI 结构、状态保持与布局边界。

本计划建议采用“小步整理、低风险推进”的方式：

1. 先修上下文保持与布局主次问题
2. 再增强页面结构与缩放表现
3. 最后收敛 UI 组件与编排层之间的接口边界

这样既能快速提升用户感知，也能为后续继续实现实时在线状态、消息体验优化和更成熟的聊天交互打下更稳定的基础。
