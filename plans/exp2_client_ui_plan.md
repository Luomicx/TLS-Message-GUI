# Experiment 2 Plan: Secure IM Tool - Client UI & Communication (PyQt5)

## 1. 目标与范围
本次实验聚焦客户端实现，目标是在现有服务端基础上规划并落地一个安全的网络聊天客户端。当前阶段优先完成 UI 设计与交互结构规划，后续再接入网络通信与安全能力。

本次计划覆盖：
- 客户端必做功能：登录、注销、搜索/添加好友、聊天
- 客户端选做功能：注册、敏感数据与通信加密
- PyQt5 桌面端 UI 设计
- 与当前服务端协议的兼容性分析
- 后续实现阶段的拆分建议

## 2. 与现有服务端的衔接现状
根据当前项目已有服务端实现，客户端在协议层可直接对接的能力包括：
- `LOGIN <username> <password>`：登录校验
- `LOGOUT <username>`：注销通知

当前服务端尚未完整实现的能力包括：
- 好友搜索与好友关系维护
- 注册协议
- 聊天消息转发协议
- 在线用户列表广播
- 更安全的数据包结构

因此，客户端计划需要采用“UI 先行、协议预留、逐步联调”的方式推进：
1. 先完成登录、注册、主聊天窗口、好友搜索等界面；
2. 对尚未具备后端支持的功能预留交互入口与状态提示；
3. 在后续联调阶段扩展服务端协议。

## 3. UI 设计原则
### 3.1 总体风格
客户端采用 PyQt5 实现桌面 GUI，整体风格参考 `doc/ui-template.md` 的视觉提示，但不机械照搬其中面向 Web class 的细节，转化为适合 PyQt5 的设计语言。

最终落地风格以“仿微信桌面端”为主，结合轻量毛玻璃质感：
- 整体布局接近微信：左侧会话/联系人区，右侧聊天主区
- 色彩以浅灰、白、浅绿为主，强调清爽、稳定、易读
- 登录与注册页面采用居中卡片式布局
- 交互元素统一圆角，保留一定玻璃感与阴影层次
- 图标、头像、状态点位全部预留占位，后续替换正式资源

### 3.2 从模板提炼出的 PyQt5 设计约束
适用于本项目的可执行约束：
- 容器使用大圆角卡片，避免生硬直角
- 输入框、按钮、列表项保持统一圆角、边框、悬停反馈
- 登录/注册卡片采用半透明或浅色高亮面板，增强层次感
- 聊天气泡、联系人卡片、搜索结果项保持统一间距体系
- 保证文本对比度，优先可读性，不追求过强透明效果
- 图标全部使用固定尺寸占位：16x16、20x20、24x24、40x40、48x48
- 所有页面都要考虑后续状态切换：加载中、失败、空列表、未连接

### 3.3 不直接照搬模板的部分
由于 `ui-template.md` 中包含明显偏 Web 的样式描述，且与“仿微信”目标并不完全一致，因此以下内容仅作为灵感参考，不直接使用：
- HTML/Tailwind class 写法
- 过强的 Liquid Glass 光学特效
- 与当前风格冲突的黑色硬边框模板

## 4. 页面与窗口规划

### 4.1 登录窗口
定位：应用启动后的第一个窗口。

布局建议：
- 顶部：应用 logo 占位 + 应用名称 `Secure IM`
- 中部卡片：
  - 账号输入框
  - 密码输入框
  - 记住账号复选框（可选预留）
  - 登录按钮
  - 前往注册按钮/文本按钮
- 底部：连接状态提示、错误提示文本区域

交互要求：
- 用户输入 Id 和密码后点击登录
- 登录成功后进入主聊天窗口
- 登录失败时显示明确错误信息，例如：
  - 用户名或密码错误
  - 账号被锁定
  - 无法连接服务器
- 支持回车触发登录

图标预留：
- 窗口 logo
- 用户输入框前缀图标
- 密码输入框前缀图标
- 登录按钮图标

### 4.2 注册窗口
定位：独立对话框或独立子窗口，优先建议使用模态对话框。

布局建议：
- 标题区：注册新账号
- 表单区：
  - 昵称输入框
  - 密码输入框
  - 确认密码输入框
- 操作区：
  - 注册按钮
  - 返回登录按钮
- 结果区：
  - 成功提示 / 失败提示

交互要求：
- 输入昵称和密码即可提交
- 若后端暂未支持注册协议，则先完成界面与本地校验逻辑
- 本地校验包括：
  - 昵称不能为空
  - 密码不能为空
  - 两次密码一致
- 预留“注册成功后自动回到登录页”的交互路径

图标预留：
- 注册标题图标
- 昵称图标
- 密码图标
- 确认密码图标

### 4.3 主聊天窗口
定位：登录成功后的核心界面。

推荐整体布局采用三栏中的双栏主形态：
- 左侧边栏：当前用户信息 + 导航标签 + 联系人/会话列表
- 右侧主区：聊天标题栏 + 消息展示区 + 输入区

#### 左侧区域
包含：
- 当前用户头像占位
- 当前用户昵称/Id
- 导航切换区：
  - 聊天
  - 好友
  - 搜索
- 列表区：
  - 最近会话列表或好友列表
- 底部操作区：
  - 注销/退出按钮

设计风格：
- 接近微信桌面端侧边栏
- 列表项支持选中、高亮、未读标识预留
- 头像使用圆角矩形或圆形占位

图标预留：
- 用户头像
- 导航图标
- 注销图标
- 未读消息红点/数字角标位置

#### 右侧聊天主区
包含：
- 顶部标题栏：当前聊天对象昵称、在线状态占位
- 中部消息滚动区：
  - 接收消息气泡
  - 发送消息气泡
  - 时间分隔标签
- 底部输入区：
  - 多行文本输入框
  - 发送按钮
  - 附件/表情按钮占位（先预留不实现）

交互要求：
- 选择联系人后切换聊天会话
- 支持输入并发送消息
- 消息区支持区分“自己发送”和“对方发送”
- 若未连接服务端或对方不可用，显示状态提示

样式要求：
- 自己发送的消息靠右，浅绿色气泡
- 对方消息靠左，白色或浅灰气泡
- 背景保持简洁，避免过多装饰影响阅读

### 4.4 好友搜索/添加区域
可采用以下两种方式之一：
- 嵌入主窗口中的“搜索”页签
- 独立弹窗

推荐优先方案：嵌入主窗口页签，便于保持微信式体验。

包含：
- 搜索输入框
- 搜索模式切换：
  - 按昵称模糊搜索
  - 按 Id 精准搜索
- 搜索按钮
- 结果列表
- 添加好友按钮

结果项展示建议：
- 头像占位
- 昵称
- 用户 Id
- 状态标签
- 添加按钮

交互要求：
- 用户输入内容后可发起搜索
- 若服务端未支持该协议，则保留按钮、列表与占位提示
- 搜索无结果时显示空状态

图标预留：
- 搜索图标
- 添加好友图标
- 用户头像图标

### 4.5 注销与退出交互
注销是必做功能，需要在客户端退出时通知服务端。

交互规划：
- 用户主动点击“注销/退出登录”按钮时：
  - 发送注销请求
  - 清理本地会话状态
  - 返回登录窗口
- 用户关闭主窗口时：
  - 若当前已登录，优先发送注销命令
  - 无论服务端是否响应，都继续执行本地退出流程

## 5. 页面状态设计
为保证 UI 可直接进入开发，所有主要页面都应预留以下状态：
- 初始空状态
- 加载中状态
- 成功状态
- 错误状态
- 服务器未连接状态
- 数据为空状态

具体示例：
- 聊天区无会话时：显示欢迎占位页
- 搜索结果为空时：显示“未找到匹配用户”
- 网络异常时：显示“服务器连接失败，请稍后重试”

当前阶段不要求完成最终图标设计，但应统一预留资源结构与占位方式。

建议预留内容：
- 应用 logo
- 默认用户头像
- 登录/注册输入框图标
- 左侧导航图标
- 搜索图标

## 7. 客户端功能与界面映射
### 7.1 必做功能
1. 登录
   - 对应登录窗口
   - 成功后进入主聊天窗口
   - 失败后在表单下方显示原因

2. 注销
   - 对应主窗口底部退出按钮和关闭窗口事件
   - 向服务端发送注销命令

3. 搜索/添加好友
   - 对应主窗口中的搜索页签或好友页签
   - 搜索结果列表支持添加按钮

4. 聊天
   - 对应主聊天窗口
   - 支持选择联系人并收发消息

### 7.2 选做功能
1. 注册
   - 对应注册对话框
   - 从登录页跳转进入

2. 加密
   - 不一定直接体现在单独窗口中，但要在 UI 上预留状态提示位，例如：
     - 当前连接是否安全
     - 当前消息是否加密发送

## 8. 安全性设计建议
本实验目标强调数据与通信安全，虽然当前阶段先做 UI，但应在计划中提前预留安全实现方向。

### 8.1 本地敏感数据
- 客户端不保存明文密码
- 若需要“记住账号”，只保存账号 Id，不保存明文密码
- 若后续支持自动登录，应保存受保护的令牌，而不是密码本身

### 8.2 网络通信
当前服务端使用简单文本协议，安全性较弱，后续建议逐步升级：
- 第一阶段：保留现有文本协议，快速联调
- 第二阶段：改为 JSON 行协议，减少解析歧义
- 第三阶段：对敏感字段进行加密或摘要保护
- 第四阶段：若条件允许，引入 TLS 或会话密钥机制

### 8.3 UI 上的安全反馈
建议在主界面中预留安全状态展示：
- 已连接 / 未连接
- 普通连接 / 安全连接
- 登录认证成功 / 失败

## 9. 推荐模块拆分
若后续开始真正编码，建议客户端采用如下模块结构：
- `client_app/__main__.py`：启动入口
- `client_app/app.py`：QApplication 初始化
- `client_app/network/client_controller.py`：网络通信控制
- `client_app/protocol.py`：协议封装与解析
- `client_app/ui/login_window.py`：登录窗口
- `client_app/ui/register_dialog.py`：注册对话框
- `client_app/ui/chat_window.py`：主聊天窗口
- `client_app/ui/widgets/`：头像、消息气泡、列表项等复用组件
- `client_app/assets/`：后续图标资源目录

当前阶段无需一次性全部实现，但 UI 结构应按该方向设计，避免后续重构成本过高。
